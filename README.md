# NetCop

NetCop — утилита для мониторинга сети и **шейпинга трафика на уровне процессов** под Windows. Скрытый агент на целевых машинах + центральный сервер с веб-дашбордом.

Главное отличие от наивных «ограничителей»: NetCop умеет честно резать **скачивание**, а не только отдачу.

## Почему download-шейпинг — это сложно (и как тут решено)

Большинство самодельных шейперов пытаются ограничить скачивание, **дропая входящие пакеты**. Это не работает как надо: когда пакет дошёл до твоей машины, байт **уже прошёл через WAN-канал**. Ты его выкинул — канал уже потрачен, а отправитель ещё и пришлёт пакет заново. Получается не throttle, а «loss-throttle» с двойным расходом канала.

NetCop использует три слоя, потому что одним трюком всё не покрыть:

| Слой | Что шейпит | Механизм |
|------|-----------|----------|
| **TCP-загрузка** (HTTP, лаунчеры, обновления) | download | **TCP Window Clamping** |
| **UDP / QUIC / uTP** (часть игр, ютуб, торрент-uTP) | download | Delay-очередь + token bucket |
| **Отдача** (любая) | upload | `NetQosPolicy` (Windows QoS) |

### TCP Window Clamping — сердце проекта

TCP-отправитель физически не может слать больше, чем рекламируемое получателем окно. Твоя машина — получатель, и в каждом исходящем ACK она сообщает «у меня столько-то места». NetCop перехватывает **исходящие** ACK и переписывает поле окна до значения BDP:

```
target_window = target_rate × RTT
```

Отправитель сам себя тормозит. **Байт не выходит на провод вообще** — ноль WAN-оверхеда, ноль потерь, ноль ретрансмитов. Роутер (даже самый дохлый провайдерский) вообще не в курсе, что происходит — вся магия на эндпоинте.

Window scale (множитель окна) считывается из TCP-хендшейка и учитывается при пересчёте.

### Честные ограничения

- **Window clamping применяется только к TCP.** QUIC/UDP/uTP ведут своё congestion control — для них работает delay-очередь (придержать и отпустить с пейсингом). Это мягче дропа, но байт по WAN всё же проходит.
- **RTT задаётся в конфиге** (по умолчанию 50 мс). Для задачи «не дать процессу сожрать весь канал» точность до мегабита не нужна — нужна предсказуемость. Можно покрутить `--rtt-ms`.
- Если агент не застал TCP-хендшейк (соединение уже было открыто), берётся scale = 2⁸ (дефолт Windows autotuning) — для типичных машин это точное попадание, в худшем случае окно чуть уже задуманного, но никогда не шире.
- **Это endpoint-шейпинг.** Идеальный шейпинг инбаунда живёт на роутере; здесь его нет, поэтому TCP закрывается клампом, а UDP — пейсингом.

## Возможности

- **Мониторинг:** CPU, RAM, активные соединения, топ-10 процессов по трафику.
- **Шейпинг IN+OUT:** кламп окна для скачивания (TCP) + QoS для отдачи + delay-пейсинг для UDP.
- **Массовое управление:** процессы группируются по категориям (торренты, игры, стриминг, браузеры) — лимит в один клик.
- **Priority Mode:** автоматически придушивает «жручие» категории, запоминает прежние лимиты и **восстанавливает именно их** при выключении.
- **Network Kill с авто-восстановлением:** вырубить сеть машины можно одной кнопкой, но это **деретёр, а не кирпич** — интерфейс сам поднимется через 2 минуты (или кнопкой Restore Net раньше). Застрять оффлайн невозможно.
- **Аудит-лог:** все действия пишутся в SQLite с временем, целью и source IP.
- **System Tray:** агент может жить фоном с индикатором в трее.

## Архитектура

- **Сервер (`server/`):** FastAPI + SQLite, веб-дашборд (HTML/JS/Chart.js).
  - Команды доставляются с гарантией **at-least-once**: висят в очереди, пока агент их не подтвердит (ACK). Потерянный из-за сетевого блика ответ больше не теряет команду. Команды идемпотентны, повтор безвреден.
  - Запись в SQLite вынесена в отдельный поток — хендлеры не блокируют event loop на диске.
  - Сравнение API-ключа — constant-time (`secrets.compare_digest`).
- **Агент (`agent/`):** Python (`psutil`, `pydivert`, `pystray`). Репортит метрики, тянет и **подтверждает** команды, держит watchdog на kill.

## API

Все эндпоинты под `/api/*` требуют заголовок `X-NetCop-Key`. `{hostname}` — имя машины из отчёта агента.

| Метод | Путь | Назначение |
|-------|------|-----------|
| `POST` | `/api/report` | Агент шлёт телеметрию (трафик + топ процессов). |
| `GET` | `/api/commands/{hostname}` | Агент забирает очередь команд (без удаления — до ACK). |
| `POST` | `/api/ack/{hostname}` | Агент подтверждает выполненные команды по их `id`. |
| `GET` | `/api/status` | Состояние всех агентов + флаг priority mode (для дашборда). |
| `GET` | `/api/history/{hostname}` | История трафика (≈30 мин) для графика. |
| `POST` | `/api/limit/{hostname}` | Глобальный лимит отдачи (Mbps). |
| `POST` | `/api/unlimit/{hostname}` | Снять глобальный лимит. |
| `POST` | `/api/limit_process/{hostname}` | Лимит отдачи на процесс (QoS). |
| `POST` | `/api/unlimit_process/{hostname}` | Снять лимит на процесс. |
| `POST` | `/api/full_throttle/{hostname}` | Лимит на процесс сразу IN+OUT (QoS + шейпер). |
| `POST` | `/api/full_unthrottle/{hostname}` | Снять IN+OUT лимит на процесс. |
| `POST` | `/api/kill/{hostname}` | Вырубить сетевой интерфейс (с авто-восстановлением). |
| `POST` | `/api/unkill/{hostname}` | Поднять интерфейс досрочно. |
| `POST` | `/api/priority_mode/on` | Включить priority mode (авто-душение «жручих» категорий). |
| `POST` | `/api/priority_mode/off` | Выключить и восстановить прежние лимиты. |
| `GET` | `/api/audit?hostname=&limit=&offset=` | Аудит-лог (по умолчанию 50 записей, максимум 500). |

## Установка и запуск

### Сервер

1. `cd server`
2. `pip install -r requirements.txt`
3. `python launcher.py`
   - При первом запуске сгенерируется API-ключ и напечатается в консоль крупным блоком. **Сохрани его** — он нужен агентам и для входа в дашборд.
4. Дашборд: `http://127.0.0.1:8000`

Bind-адрес и порт настраиваются через `server/config.json` или аргументами:
```
python launcher.py --host 0.0.0.0 --port 8000
```
По умолчанию слушает только `127.0.0.1`. Чтобы агенты с других машин могли подключаться — поставь `--host 0.0.0.0` и пропиши `allowed_origins` в config.json.

Пример `server/config.json` (генерируется при первом запуске, ключ — автоматически):
```json
{
    "api_key": "<сгенерированный-ключ-32-символа>",
    "host": "0.0.0.0",
    "port": 8000,
    "allowed_origins": ["http://192.168.1.100:8000"],
    "priority_profile": {
        "torrent":   { "in_kbps": 128, "out_kbps": 64 },
        "gaming":    { "in_kbps": 256, "out_kbps": 128 },
        "streaming": { "in_kbps": 256, "out_kbps": 128 }
    }
}
```
`allowed_origins` пустой → CORS работает в режиме без credentials (нормально, т.к. авторизация идёт через заголовок, а не куки). Если указать origin'ы — включается credentialed CORS только для них. `config.json` в `.gitignore` — не коммить его (там ключ).

### Агент

1. `cd agent`
2. Запусти `install.bat` (от администратора). Скрипт спросит:
   - URL сервера
   - **API-ключ (обязательно)** — тот, что напечатал сервер
   - Включить ли WinDivert-шейпер (и RTT для клампа)
   - Запускать ли в трее
3. Агент стартует автоматически. **Только от имени администратора.**

## Troubleshooting: «лимит не работает»

**Upload (QoS) не режется:**
- На не-доменных машинах policy-based QoS по умолчанию **выключен**. Агент при старте сам ставит ключ реестра `HKLM\SYSTEM\CurrentControlSet\Services\Tcpip\QoS → "Do not use NLA" = "1"`, но после первой установки нужен **один перезапуск Windows** — без него NetQosPolicy-политики создаются, но не применяются.
- Проверить, что политики вообще создались: `Get-NetQosPolicy` в PowerShell от администратора. Ошибки PowerShell теперь пишутся в `agent.log`.

**Download (кламп) не режется:**
- Шейпер должен быть включён (`--enable-shaper`), `pydivert` установлен, а `WinDivert.dll`/`WinDivert64.sys` лежать рядом с exe.
- Кламп ограничивает «окно = rate × RTT», поэтому реальная скорость = окно / **реальный** RTT. Если тестируешь по близкому серверу (RTT 3–5 мс при дефолтных 50 мс) — скорость будет в ~10 раз выше лимита. Это не баг клампа, а физика: либо тестируй по далёкому серверу, либо поставь `--rtt-ms` ближе к реальному RTT.
- Кламп действует только на **новые** данные: уже летящий поток подожмётся в течение нескольких секунд.

**Агент умер / отвалился:**
- Смотри `agent.log` — все ошибки цикла теперь логируются с traceback, цикл не падает от одиночного сбоя.
- После Kill Net агент «offline» на дашборде — это норма: интерфейс лежит до Restore Net или авто-восстановления (2 мин).

## Безопасность

- Связь агент↔сервер защищена общим секретом в заголовке `X-NetCop-Key`.
- **Важно:** ключ ходит **плейнтекстом**. В пределах LAN это приемлемо, но в недоверенной сети подними сервер за TLS-прокси (nginx/caddy) — иначе ключ можно снять снифером.
- Дефолтный ключ `"secret"` и ключи короче 16 символов отклоняются — сервер не стартует.

## Что изменилось в этой версии

Переписано после ревью. Основное:

1. **Download-шейпинг теперь настоящий** — добавлен TCP window clamping. Старый код дропал входящие пакеты (loss-throttle с двойным расходом канала); QoS вообще режет только отдачу. Старый шейпер стал UDP-веткой.
2. **Token bucket больше не дедлочится** на низких лимитах. Был `burst = rate × 0.5` → ниже ~24 кбит/с burst падал под один MTU и `consume()` зависал намертво. Теперь burst floored под 64 КиБ (заодно переживает LRO/RSC super-segments).
3. **Команды не теряются.** Очередь чистится только после ACK от агента, а не сразу на выдаче. Потерянный ответ → команда переотправится.
4. **Двойное списание токенов убрано.** Per-process и global бакеты теперь координируются через peek/commit — пакет либо списывается у обоих, либо ни у кого.
5. **Network Kill безопасен.** Добавлены unkill + watchdog авто-восстановления. Раньше kill был билетом в один конец — только ногами к машине.
6. **Безопасность:** constant-time сравнение ключа, исправлена невалидная пара `CORS allow_origins=["*"] + allow_credentials=True`.
7. **Прочее:** определение интерфейса по дефолтному маршруту (а не первому «up», которым мог оказаться виртуальный VPN/Hyper-V адаптер), запись в SQLite в отдельном потоке, TTL на висящие команды.
